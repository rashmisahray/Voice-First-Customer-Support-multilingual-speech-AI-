# ADR-001: Choosing Whisper for Initial Speech Recognition

## Context

Vani is a voice-first Customer Support AI Agent that must transcribe spoken audio input from a user's microphone or uploaded audio files into structured text. This transcription is then analyzed by NLU intent detection and entity extraction. The requirements demand English support for Week 2, with the architectural readiness to scale to Hindi and Hinglish (code-mixed speech) in subsequent weeks. High accuracy, robust handling of diverse accents, local CPU execution, and low runtime latency are crucial parameters.

## Decision

We chose **Whisper** (specifically implemented via the **faster-whisper** engine) as our initial Speech Recognition model, using the optimized `tiny.en` model running locally on the CPU with INT8 quantization.

`faster-whisper` is a re-implementation of OpenAI's Whisper model using CTranslate2, which is a fast inference engine for Transformer models. It is up to 4x faster than standard PyTorch-based OpenAI Whisper while using significantly less memory.

To support offline and lightweight installation on diverse developer machines without requiring system-wide FFmpeg binaries, we implemented a custom in-memory WAV decoder in Python. This custom helper converts incoming WAV PCM data directly into the 16kHz float32 NumPy format that `faster-whisper` expects.

## Consequences

### Advantages
- **High Accuracy & Accent Robustness**: Whisper is trained on a massive dataset of multilingual audio and excels at transcribing natural spoken English and varying accents.
- **Optimized CPU Performance**: Using CTranslate2 with `compute_type="int8"` allows fast transcriptions on standard consumer CPUs without requiring a dedicated GPU.
- **Multilingual Roadmap**: The model can be swapped from `tiny.en` to a multilingual variant (e.g. `tiny` or `base`) to support Hindi and Hinglish without rewriting the ASR codebase.
- **Zero-Dependency Fallback**: The backend decodes WAV audio directly in Python using standard libraries, removing external binary dependencies like FFmpeg for core WAV features.

### Disadvantages
- **Model Download Size**: Even the smallest `tiny.en` model requires downloading ~75MB of model files on the first run.
- **Cold-Start Latency**: The first transcription request in a session takes slightly longer due to lazy loading of the model files into memory.

### Trade-offs
- We trade off the convenience of a hosted API (like OpenAI's Whisper API) for a fully local execution model that requires zero external API keys and guarantees high data privacy and reduced long-term costs.

## Alternatives considered

1. **Google Speech-to-Text API**:
   - *Why rejected*: It is a cloud-only, paid service requiring Google Cloud service accounts and API key setups. It does not fit the requirement for an easy-to-use local development experience that can run offline.
2. **PocketSphinx (local)**:
   - *Why rejected*: While extremely fast and lightweight, pocket-sphinx has poor accuracy on conversational speech, modern accent variations, and mixed languages.
3. **OpenAI Whisper (standard PyTorch package)**:
   - *Why rejected*: It requires installing PyTorch (which is ~2GB+ to download) and is significantly slower on CPU execution compared to CTranslate2.
